## Критичные текущие замечания

### 1. Build требует зафиксированного последовательного режима

Файл:
- `global.json`

Проблема:
- корневой `global.json` теперь валиден и выбирает .NET 8 SDK;
- `dotnet build` подтвержден для всех API-проектов через `-m:1`;
- параллельный restore/build без `-m:1` может падать без явных compile errors;
- без writable `DOTNET_CLI_HOME` в sandbox/CI-like окружении `dotnet` может падать на first-time setup.

Почему это плохо:
- если не зафиксировать команду, CI/runbook будут зависеть от случайного MSBuild parallel behavior;
- команда сборки должна быть явно описана в runbook.

### 2. Telemetry batch-size validator сейчас инвертирован

Файл:
- `src/Services/DeviceService/Device.Application/DTOs/Validators/TelemetryBatchRequestValidator.cs`

Проблема:
- правило написано как `.Must(x => x.Count > 50)`;
- сообщение при этом говорит `"Maximum batch size is 50 items."`;
- фактически нормальная пачка на 1-50 элементов будет считаться невалидной.

Почему это плохо:
- первый smoke с МК/эмулятором может упасть на валидной телеметрии;
- дефект находится до публикации `TelemetryReportedFromHardwareEvent`, поэтому downstream-сценарий не стартует.

### 3. Text ошибок telemetry validation требует чистки

Файлы:
- `src/Services/DeviceService/Device.Application/DTOs/Validators/TelemetryItemRequestValidator.cs`
- `README.md`

Проблема:
- для `ExternalMessageId` используется сообщение `"SensorId must not be empty."`;
- README/docs должны отражать фактические validation errors после исправления batch-size правила.

Почему это плохо:
- диагностика для МК будет вводить в заблуждение.

### 4. Physical MCU runbook не зафиксирован

Файлы:
- `docker-compose.yml`
- `src/Gateway/ApiGateway/appsettings.json`

Проблема:
- через gateway нужен JWT;
- direct `device-api` уже опубликован на `5237`;
- lab mode уже описан в `README.md`, но фактический smoke-flow еще не пройден.

Почему это плохо:
- endpoint и контракт описаны, но подключение физического МК еще не подтверждено runtime-сценарием.

## Security and Architecture

### 5. Refresh token reuse policy не доведена

Файлы:
- `src/Services/IdentityService/IdentityService.Domain/Entities/RefreshTokenEntity.cs`
- `src/Services/IdentityService/IdentityService.Infrastructure/Repositories/RefreshTokenRepository.cs`

Проблема:
- raw storage заменен на hash;
- при reuse старого refresh token-а нет отдельной политики компрометации token family/session family.

Минимальный стандарт:
- продумать reuse detection;
- определить family/session revocation;
- описать TTL и logout semantics в docs.

### 6. Cookie settings пока локальные

Файл:
- `src/Services/IdentityService/IdentityService/Controllers/AuthController.cs`

Проблема:
- auth cookie выставляется с `Secure = false`.

Почему это плохо:
- это нормально для локального HTTP;
- для production HTTPS нужно `Secure = true` и осознанная SameSite/CORS policy.

### 7. Subscription downgrade остается локальным

Файл:
- `src/Services/IdentityService/IdentityService.Application/Services/SubscriptionExpiredChecker.cs`

Проблема:
- downgrade подписки не публикует событие.

Почему это плохо:
- downstream-сервисы не узнают об изменении лимитов/прав.

## Code Style and Maintainability

### 8. Naming hygiene все еще слабая

Примеры:
- `AquariumCreatedEvend`
- `AquarimUdatedEvent`
- `AquriumId`
- `Device.Infrastrucrute`
- `ControllerRegistredResponseDto`

Почему это плохо:
- такие опечатки уже приводили к ошибкам в imports;
- стоимость поддержки и ревью растет.

### 9. В репозитории лежат IDE/build/temp артефакты

Примеры:
- `.vs`
- `.idea`
- `.dotnet`
- `.dotnet-home`
- `build.log`
- `infra-build*.log`
- `restore-diag.log`
- `*.Backup.tmp`

Почему это плохо:
- шумит diff;
- ухудшает воспроизводимость;
- смешивает исходники и локальное состояние машины.

## Минимальный стандарт перед следующим ревью

1. Команда `dotnet build ... -m:1` зафиксирована в runbook/README.
2. Исправлено batch-size правило в `TelemetryBatchRequestValidator`.
3. Direct MCU lab mode подтвержден runtime-smoke-ом.
4. Validation errors для telemetry ingress согласованы с README/docs.
5. Пройден `docker compose up --build`.
6. Пройден smoke: register/login -> create controller/sensor -> telemetry -> telemetry saved -> control reaction.
7. Refresh token reuse/family revocation policy описана или реализована.
